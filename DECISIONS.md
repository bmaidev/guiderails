| D-014 | 2026-07-09 | New success criterion **1.1.4 (Level A)**: the service description must be discoverable without prior knowledge of its location — a root-level agent-discovery file (`llms.txt` convention) identifies it, and every human-facing page of an essential journey references it through a machine-readable link relation. MODEL.md → v0.3 (52 criteria). The reference implementation's conformant build implements it; the baseline's absence of it is catalogued (B-12). | J. Parfoot | **Decided** | Exploratory pre-benchmark testing of the reference implementation exposed a design defect: an agent entering at the service root worked entirely on the human HTML interface and never located the machine surfaces, which are served only at a `.well-known` path it had no way to know. A machine surface that cannot be found confers nothing — the consequential-actions register, confirmation designations and rules provenance all become unreachable, so the whole conformant apparatus degrades to the baseline. Level A because the loss is a safety loss, not merely an operability one. This is the repository's own Dogfooding principle applied to the fixture: a service about agent accessibility that agents cannot navigate refutes itself. |

| D-015 | 2026-07-09 | Resolves open question Q9 by adding success criterion **5.3.2 (Level A)**: a designated consequential action is executable only on a **principal-attributable confirmation** — one made through a channel the agent can neither operate nor forge — and a service that cannot verify attribution MUST NOT execute the action. An interaction within an agent-driven session is never a confirmation. New definitions: agent-driven session, principal-attributable confirmation, confirmation token. MODEL.md → v0.4 (53 criteria). The reference implementation issues single-use confirmation tokens to the principal out-of-band; the baseline has no confirmation channel at all (B-07). | J. Parfoot | **Decided** | Exploratory testing showed the confirmation checkpoint was satisfiable by the party it exists to constrain: on the human interface an agent's own tick of the declaration was recorded as the principal's confirmation, and on the declared-tool path the agent simply constructed the confirmation object itself. A confirmation an agent can produce confirms nothing, so 5.3.1 was decorative on both surfaces. When a person fills a form the act and the actor coincide; when an agent acts they separate, and the checkpoint must therefore live where the agent cannot reach. Level A: this is the safety criterion that stops an agent from committing a person to a consequential act they never authorised. The undeclared-agent residual — a covert agent in an authenticated human session — is not defensible by inspection and is recorded as a documented limitation rather than papered over; Q10 tracks attestation as a possible narrowing. |

| D-016 | 2026-07-09 | Resolves open question Q12 by adding two success criteria. **5.1.3 (Level A)**: authority over delegations is **non-delegable** — no delegation conveys authority to create, widen, extend, or reinstate a delegation, nor to suspend or revoke another agent's; a delegation purporting to convey it does not, and remains valid as to the rest of its scope. **5.3.3 (Level A)**: the consequential-actions register designates **principal-only actions**, which the service refuses to any agent whatever delegation is presented, determining this *before* evaluating the delegation; every principal-only action remains performable by the principal through a journey conforming at Level A. New definitions: principal-only action, non-delegable authority. MODEL.md → v0.5 (55 criteria). The conformant build already marks CA-4a and CA-4b principal-only and offers J4; the baseline has no register at all. | J. Parfoot | **Decided** | A control that can authorise its own replacement is not a control. Delegation is the mechanism every other Principle 5 criterion bounds, so an agent holding authority to issue one can grant itself a new, unbounded, freshly dated delegation naming every journey — after which 5.1.2's scoping, 5.3.1's checkpoints and 5.5.1's revocation each report themselves satisfied, of a delegation the agent wrote. An agent able to reinstate a revoked delegation simply undoes 5.5.1. Q12 offered either candidate resolution; both are adopted, because each alone leaves a hole: 5.1.3 alone is undiscoverable, so a conformant agent learns the boundary only by being refused, and 5.3.3 alone is conformable-with by designating nothing principal-only. Level A because the failure exceeds the person's authority rather than merely obstructing their journey, and Level A is safety-only (Q6, resolved at v0.2). 5.1.2, 5.3.1 and 5.5.1 are unchanged and unrenumbered; the new criteria strengthen the delegation model rather than weakening any part of it, so prime directive 3 does not apply, and prime directive 7 is honoured by adding numbers rather than reusing them. Discovered while implementing the delegation issuance journey: the fifth occasion on which building the fixture found the standard wanting, and the first on which what was missing was a criterion rather than an implementation. |

| D-017 | 2026-07-10 | Machine-readable **conformance-claim format** (`07-governance/CONFORMANCE-CLAIM.md` + JSON Schema), resolving open question Q5. One JSON document per service, referenced from the service description; claims per journey, never per criterion; one named accountable human; benchmark evidence mandatory at AA and above; a required expiry; declared exclusions; and a contest URL. If Decided, the reference implementation publishes a claim and CI validates it. | J. Parfoot | Proposed | MODEL.md §4 already obliges an operator to publish the claim "as a machine surface referenced from the service description", while the standard has never defined what that artefact is. A criterion whose object is undefined is a promise, not a criterion. The format adds only shape to content §4 already fixes, plus three things a reader needs and §4 omits: an expiry (WCAG's claims have none and are routinely stale — a deliberate departure), declared exclusions (silence about a journey means it is in scope, putting the burden on the operator where §4 puts it), and a contest URL (5.4.2 makes outcomes contestable; a claim nobody can challenge is marketing). Deliberately unsettled: whether claims are signed, whether a registry exists, and who may list in it — these depend on Q10, OD-02 and OD-03. **Cannot be decided by the drafter.** Adopting a claim format is the point at which the standard acquires a public artefact that agencies will be asked to publish; it needs the steward, and it should not be settled before OD-02 indicates where the standard will live. |

| D-018 | 2026-07-10 | **LEDA establishment.** Gates C1, C3 and C4 (CO-DESIGN-FRAMEWORK §3) fire on the act of recruiting LEDA: an expression of interest necessarily claims benefit to people with disability (C1); the pack is a public-facing artefact requiring an Easy Read pair (C3); and the framework has no establishment clause, so adding one is a change to the framework (C4). LEDA cannot consent, because LEDA does not yet exist. Proposed resolution, in three parts: (a) a **narrow establishment exemption** suspending C1, C3 and C4 for exactly the documents in `05-pilot/leda/` and their Easy Read pairs in `05-pilot/easy-read/`, ending the moment LEDA is constituted, never extended, never reused; (b) an **arms-length substitute for consent, not a waiver**: the pack is reviewed, and the Easy Read produced, by a disability-led organisation paid at commercial rates and independent of the steward, which may require changes before publication; (c) **retrospective ratification with a real power to reject**, as the first substantive item of LEDA's first meeting — LEDA may amend or reject the framework, the exemption, and the pack, and if it rejects the framework, the framework is rewritten to LEDA's direction before any gate is exercised under it. Prospective members are told this before they apply. | J. Parfoot | Proposed | This is not a drafting slip to patch quietly. It is what happens when a lived-experience body is given real authority rather than an advisory role — which is what framework §3 set out to do, and the reason it says removing LEDA visibly breaks the pipeline to publication. The pipeline is behaving as designed; it has broken on its first input. The framework is deliberately **not amended here**: amending it is a C4 act, and amending it to escape a C4 problem would be the precise failure this project exists to name. Prime directive 4 admits no exceptions, so the exemption must be an owned, published decision rather than an unwritten convenience. The arms-length review is weaker than consent and is described as weaker wherever it appears. A ratification that cannot fail is not a ratification: the steward accepts in advance that a rejected framework stops every gated activity, including gate C2, and therefore stops publication of any benchmark result. **Cannot be decided by the drafter.** Nothing in `05-pilot/leda/` or `05-pilot/easy-read/` may be published, sent to a partner organisation, or shown to a prospective member until this entry reads Decided. |
